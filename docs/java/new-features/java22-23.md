## JDK 23

### JEP 455: Kiểu nguyên thủy trong Pattern, instanceof và switch (bản xem trước)

Trước JEP 455, `instanceof` chỉ hỗ trợ kiểu tham chiếu, và nhãn `case` trong biểu thức và câu lệnh `switch` chỉ có thể sử dụng số nguyên literal, hằng số enum và chuỗi literal.

Trong tính năng xem trước của JEP 455, `instanceof` và `switch` hỗ trợ đầy đủ tất cả các kiểu nguyên thủy, bao gồm `byte`, `short`, `char`, `int`, `long`, `float`, `double`, `boolean`.

```java
// 传统写法
if (i >= -128 && i <= 127) {
    byte b = (byte)i;
    ... b ...
}

// 使用 instanceof 改进
if (i instanceof byte b) {
    ... b ...
}

long v = ...;
// 传统写法
if (v == 1L) {
    // ...
} else if (v == 2L) {
    // ...
} else if (v == 10_000_000_000L) {
    // ...
}

// 使用 long 类型的 case 标签
switch (v) {
    case 1L:
        // ...
        break;
    case 2L:
        // ...
        break;
    case 10_000_000_000L:
        // ...
        break;
    default:
        // ...
}
```

### JEP 466: Class File API (lần xem trước thứ hai)

Class File API được xem trước lần đầu trong JDK 22, được đề xuất bởi [JEP 457](https://openjdk.org/jeps/457).

Mục tiêu của Class File API là cung cấp một bộ API chuẩn hóa để phân tích, tạo và chuyển đổi các tệp lớp Java, thay thế sự phụ thuộc vào các thư viện bên thứ ba (như ASM) trong việc xử lý tệp lớp.

```java
// 创建一个 ClassFile 对象，这是操作类文件的入口。
ClassFile cf = ClassFile.of();
// 解析字节数组为 ClassModel
ClassModel classModel = cf.parse(bytes);

// 构建新的类文件，移除以 "debug" 开头的所有方法
byte[] newBytes = cf.build(classModel.thisClass().asSymbol(),
        classBuilder -> {
            // 遍历所有类元素
            for (ClassElement ce : classModel) {
                // 判断是否为方法 且 方法名以 "debug" 开头
                if (!(ce instanceof MethodModel mm
                        && mm.methodName().stringValue().startsWith("debug"))) {
                    // 添加到新的类文件中
                    classBuilder.with(ce);
                }
            }
        });
```

### JEP 469：Vector API (lần ươm tạo thứ tám)

Tính toán vector bao gồm một chuỗi các phép toán trên các vector. Vector API được dùng để biểu diễn các phép tính vector, có thể được biên dịch đáng tin cậy tại runtime thành các lệnh vector tối ưu trên kiến trúc CPU được hỗ trợ, đạt hiệu suất vượt trội so với các phép tính scalar tương đương.

Mục tiêu của Vector API là cung cấp cho người dùng cách biểu đạt phạm vi rộng các phép tính vector một cách đơn giản, dễ sử dụng và độc lập với nền tảng.

Đây là phép tính scalar đơn giản trên các phần tử mảng:

```java
void scalarComputation(float[] a, float[] b, float[] c) {
   for (int i = 0; i < a.length; i++) {
        c[i] = (a[i] * a[i] + b[i] * b[i]) * -1.0f;
   }
}
```

Đây là phép tính vector tương đương sử dụng Vector API:

```java
static final VectorSpecies<Float> SPECIES = FloatVector.SPECIES_PREFERRED;

void vectorComputation(float[] a, float[] b, float[] c) {
    int i = 0;
    int upperBound = SPECIES.loopBound(a.length);
    for (; i < upperBound; i += SPECIES.length()) {
        // FloatVector va, vb, vc;
        var va = FloatVector.fromArray(SPECIES, a, i);
        var vb = FloatVector.fromArray(SPECIES, b, i);
        var vc = va.mul(va)
                   .add(vb.mul(vb))
                   .neg();
        vc.intoArray(c, i);
    }
    for (; i < a.length; i++) {
        c[i] = (a[i] * a[i] + b[i] * b[i]) * -1.0f;
    }
}
```

### JEP 471：Đánh dấu lỗi thời các phương thức truy cập bộ nhớ trong sun.misc.Unsafe

JEP 471 đề xuất đánh dấu lỗi thời các phương thức truy cập bộ nhớ trong `sun.misc.Unsafe`, các phương thức này sẽ bị xóa trong các phiên bản tương lai.

Các phương thức không an toàn này đã có các giải pháp thay thế an toàn và hiệu quả:

- `java.lang.invoke.VarHandle`: được giới thiệu trong JDK 9 (JEP 193), cung cấp cách thao tác bộ nhớ heap an toàn và hiệu quả, bao gồm các trường của đối tượng, trường static của lớp và các phần tử mảng.
- `java.lang.foreign.MemorySegment`: được giới thiệu trong JDK 22 (JEP 454), cung cấp cách truy cập bộ nhớ ngoài heap an toàn và hiệu quả, đôi khi hoạt động cùng với `VarHandle`.

Hai lớp này là các thành phần cốt lõi của Foreign Function & Memory API, được dùng để quản lý và thao tác bộ nhớ ngoài heap. Foreign Function & Memory API đã trở thành tính năng chuẩn chính thức trong JDK 22.

```java
import jdk.incubator.foreign.*;
import java.lang.invoke.VarHandle;

// 管理堆外整数数组的类
class OffHeapIntBuffer {

    // 用于访问整数元素的VarHandle
    private static final VarHandle ELEM_VH = ValueLayout.JAVA_INT.arrayElementVarHandle();

    // 内存管理器
    private final Arena arena;

    // 堆外内存段
    private final MemorySegment buffer;

    // 构造函数，分配指定数量的整数空间
    public OffHeapIntBuffer(long size) {
        this.arena  = Arena.ofShared();
        this.buffer = arena.allocate(ValueLayout.JAVA_INT, size);
    }

    // 释放内存
    public void deallocate() {
        arena.close();
    }

    // 以volatile方式设置指定索引的值
    public void setVolatile(long index, int value) {
        ELEM_VH.setVolatile(buffer, 0L, index, value);
    }

    // 初始化指定范围的元素为0
    public void initialize(long start, long n) {
        buffer.asSlice(ValueLayout.JAVA_INT.byteSize() * start,
                       ValueLayout.JAVA_INT.byteSize() * n)
              .fill((byte) 0);
    }

    // 将指定范围的元素复制到新数组
    public int[] copyToNewArray(long start, int n) {
        return buffer.asSlice(ValueLayout.JAVA_INT.byteSize() * start,
                              ValueLayout.JAVA_INT.byteSize() * n)
                     .toArray(ValueLayout.JAVA_INT);
    }
}
```

### JEP 476：Khai báo import module (bản xem trước)

Khai báo import module cho phép nhập gọn tất cả các gói được xuất của toàn bộ một module trong code Java mà không cần khai báo từng import gói riêng lẻ. Tính năng này đơn giản hóa việc tái sử dụng các thư viện module hóa, đặc biệt khi sử dụng nhiều module, tránh phải khai báo rất nhiều import gói, giúp nhà phát triển truy cập thư viện bên thứ ba và các lớp cơ bản Java thuận tiện hơn.

Tính năng này đặc biệt hữu ích cho người mới học và phát triển nguyên mẫu, vì nó không yêu cầu nhà phát triển phải module hóa code của mình, đồng thời vẫn giữ tính tương thích với cách import truyền thống, nâng cao hiệu quả phát triển và khả năng đọc code.

```java
// 导入整个 java.base 模块，开发者可以直接访问 List、Map、Stream 等类，而无需每次手动导入相关包
import module java.base;

public class Example {
    public static void main(String[] args) {
        String[] fruits = { "apple", "berry", "citrus" };
        Map<String, String> fruitMap = Stream.of(fruits)
            .collect(Collectors.toMap(
                s -> s.toUpperCase().substring(0, 1),
                Function.identity()));

        System.out.println(fruitMap);
    }
}
```

### JEP 477：Lớp không tên và phương thức main của instance (lần xem trước thứ ba)

Tính năng này chủ yếu đơn giản hóa khai báo phương thức `main`. Đối với người mới học Java, khai báo phương thức `main` này giới thiệu quá nhiều khái niệm cú pháp Java, không thuận tiện cho người mới bắt đầu.

Định nghĩa phương thức `main` trước khi sử dụng tính năng này:

```java
public class HelloWorld {
    public static void main(String[] args) {
        System.out.println("Hello, World!");
    }
}
```

Định nghĩa phương thức `main` sau khi sử dụng tính năng mới:

```java
class HelloWorld {
    void main() {
        System.out.println("Hello, World!");
    }
}
```

Đơn giản hóa hơn nữa (lớp không tên cho phép bỏ qua tên lớp)

```java
void main() {
   System.out.println("Hello, World!");
}
```

### JEP 480：Structured Concurrency (lần xem trước thứ ba)

Java 19 đã giới thiệu Structured Concurrency, một phương pháp lập trình đa luồng nhằm đơn giản hóa lập trình đa luồng thông qua Structured Concurrency API, không phải để thay thế `java.util.concurrent`, hiện đang ở giai đoạn ươm tạo.

Structured Concurrency coi nhiều tác vụ chạy trong các luồng khác nhau như một đơn vị công việc duy nhất, từ đó đơn giản hóa xử lý lỗi, tăng độ tin cậy và nâng cao khả năng quan sát. Tức là, Structured Concurrency bảo toàn tính dễ đọc, dễ bảo trì và quan sát được của code đơn luồng.

API cơ bản của Structured Concurrency là [`StructuredTaskScope`](https://download.java.net/java/early_access/loom/docs/api/jdk.incubator.concurrent/jdk/incubator/concurrent/StructuredTaskScope.html). `StructuredTaskScope` hỗ trợ chia nhỏ tác vụ thành nhiều tác vụ con concurrent, thực thi trong luồng riêng của chúng, và các tác vụ con phải hoàn thành trước khi tác vụ chính tiếp tục.

Cách dùng cơ bản của `StructuredTaskScope`:

```java
    try (var scope = new StructuredTaskScope<Object>()) {
        // 使用fork方法派生线程来执行子任务
        Future<Integer> future1 = scope.fork(task1);
        Future<String> future2 = scope.fork(task2);
        // 等待线程完成
        scope.join();
        // 结果的处理可能包括处理或重新抛出异常
        ... process results/exceptions ...
    } // close
```

Structured Concurrency rất phù hợp với virtual thread, virtual thread là các luồng nhẹ do JDK triển khai. Nhiều virtual thread chia sẻ cùng một luồng hệ điều hành, cho phép tạo số lượng rất lớn virtual thread.

### JEP 481：Scoped Values (lần xem trước thứ ba)

Scoped Values có thể chia sẻ dữ liệu bất biến trong cùng một luồng và giữa các luồng, tốt hơn so với biến thread-local, đặc biệt khi sử dụng số lượng lớn virtual thread.

```java
final static ScopedValue<...> V = new ScopedValue<>();

// In some method
ScopedValue.where(V, <value>)
           .run(() -> { ... V.get() ... call methods ... });

// In a method called directly or indirectly from the lambda expression
... V.get() ...
```

Scoped Values cho phép chia sẻ dữ liệu an toàn và hiệu quả giữa các thành phần trong chương trình lớn mà không cần dùng đến tham số phương thức.

### JEP 482：Thân constructor linh hoạt (lần xem trước thứ hai)

Tính năng này ban đầu được đề xuất trong JDK 22 bởi [JEP 447: Statements before super(...) (Preview)](https://openjdk.org/jeps/447).

Java yêu cầu trong constructor, lời gọi `super(...)` hoặc `this(...)` phải xuất hiện như câu lệnh đầu tiên. Điều này có nghĩa là chúng ta không thể khởi tạo trực tiếp các trường trong constructor của lớp con trước khi gọi constructor của lớp cha.

Thân constructor linh hoạt giải quyết vấn đề này, cho phép viết các câu lệnh trong thân constructor trước khi gọi `super(..)` hoặc `this(..)`, các câu lệnh này có thể khởi tạo trường nhưng không thể tham chiếu đến instance đang được tạo. Điều này ngăn chặn việc các trường của lớp con chưa được khởi tạo đúng cách khi gọi phương thức lớp con trong constructor của lớp cha, tăng cường độ tin cậy của việc xây dựng lớp.

Tính năng này giải quyết vấn đề trước đây khi cú pháp Java hạn chế cách tổ chức code constructor, cho phép nhà phát triển biểu đạt hành vi constructor một cách tự do và tự nhiên hơn, chẳng hạn như trực tiếp thực hiện kiểm tra tham số, chuẩn bị và chia sẻ trong constructor mà không cần dựa vào phương thức hỗ trợ hay constructor phụ, cải thiện khả năng đọc và bảo trì code.

```java
class Person {
    private final String name;
    private int age;

    public Person(String name, int age) {
        if (age < 0) {
            throw new IllegalArgumentException("Age cannot be negative.");
        }
        this.name = name; // 在调用父类构造函数之前初始化字段
        this.age = age;
        // ... 其他初始化代码
    }
}

class Employee extends Person {
    private final int employeeId;

    public Employee(String name, int age, int employeeId) {
        this.employeeId = employeeId; // 在调用父类构造函数之前初始化字段
        super(name, age); // 调用父类构造函数
        // ... 其他初始化代码
    }
}
```

## JDK 22

### JEP 423：G1 Garbage Collector Region Pinning

JEP 423 đề xuất triển khai chức năng Region Pinning trong G1 garbage collector, nhằm giảm thiểu các vấn đề độ trễ do các vùng critical của Java Native Interface (JNI) gây ra.

Các đối tượng trong vùng critical của JNI không thể di chuyển trong quá trình garbage collection, vì vậy trước đây G1 giải quyết vấn đề này bằng cách vô hiệu hóa garbage collection, dẫn đến việc các luồng bị chặn và độ trễ nghiêm trọng. Bằng cách giới thiệu cơ chế region pinning trong vùng old generation và young generation của G1, cho phép cố định vùng bộ nhớ nơi đối tượng nằm trong vùng critical, đồng thời tiếp tục thu hồi các vùng chưa được ghim, tránh được nhu cầu vô hiệu hóa garbage collection. Cải tiến này giúp giảm đáng kể độ trễ, nâng cao throughput và sự ổn định của hệ thống khi tương tác với JNI.

### JEP 454：Foreign Function and Memory API

Các chương trình Java có thể tương tác với code và dữ liệu bên ngoài Java runtime thông qua API này. Bằng cách gọi hiệu quả các hàm ngoài (tức là code ngoài JVM) và truy cập an toàn bộ nhớ ngoài (tức là bộ nhớ không được JVM quản lý), API này cho phép các chương trình Java gọi thư viện native và xử lý dữ liệu native mà không nguy hiểm và dễ vỡ như JNI.

Foreign Function and Memory API đã trải qua vòng ươm tạo đầu tiên trong Java 17, được đề xuất bởi [JEP 412](https://openjdk.java.net/jeps/412). Lần ươm tạo thứ hai trong Java 18, được đề xuất bởi [JEP 419](https://openjdk.org/jeps/419). Lần xem trước đầu tiên trong Java 19, được đề xuất bởi [JEP 424](https://openjdk.org/jeps/424). Lần xem trước thứ hai trong JDK 20, được đề xuất bởi [JEP 434](https://openjdk.org/jeps/434). Lần xem trước thứ ba trong JDK 21, được đề xuất bởi [JEP 442](https://openjdk.org/jeps/442).

Cuối cùng, tính năng này đã được chính thức trong JDK 22.

Trong [Tổng quan tính năng mới Java 19](./java19.md), tôi đã giới thiệu chi tiết về Foreign Function and Memory API, nên ở đây sẽ không giới thiệu thêm.

### JEP 456：Unnamed Patterns and Variables

Unnamed Patterns and Variables được đề xuất xem trước trong JDK 21 bởi [JEP 443](https://openjdk.org/jeps/443), và đã được chính thức trong JDK 22.

Để biết thêm chi tiết về tính năng mới này, bạn có thể xem phần giới thiệu trong bài viết [Tổng quan tính năng mới Java 21 (Quan trọng)](./java21.md).

### JEP 458：Khởi chạy chương trình nguồn đa tệp

Java 11 đã giới thiệu [JEP 330：Khởi chạy chương trình nguồn đơn tệp](https://openjdk.org/jeps/330), nâng cao chức năng của trình khởi chạy `java`, cho phép chạy trực tiếp một tệp nguồn Java đơn lẻ. Thông qua lệnh `java HelloWorld.java`, Java có thể biên dịch ẩn code nguồn trong bộ nhớ và thực thi ngay lập tức mà không cần tạo tệp `.class` trên đĩa. Điều này đơn giản hóa quy trình làm việc của nhà phát triển khi viết các công cụ nhỏ hoặc học Java, tránh bước biên dịch thủ công.

Giả sử tệp `Prog.java` khai báo hai lớp:

```java
class Prog {
    public static void main(String[] args) { Helper.run(); }
}

class Helper {
    static void run() { System.out.println("Hello!"); }
}
```

Lệnh `java Prog.java` sẽ biên dịch hai lớp trong bộ nhớ và thực thi phương thức `main` của lớp đầu tiên được khai báo trong tệp đó.

Cách này có một hạn chế: tất cả code nguồn của chương trình phải đặt trong một tệp `.java`.

[JEP 458：Khởi chạy chương trình nguồn đa tệp](https://openjdk.org/jeps/458) là phần mở rộng chức năng của JEP 330, cho phép chạy trực tiếp các chương trình bao gồm nhiều tệp nguồn Java mà không cần bước biên dịch rõ ràng.

Giả sử một thư mục có hai tệp nguồn Java `Prog.java` và `Helper.java`, mỗi tệp khai báo một lớp:

```java
// Prog.java
class Prog {
    public static void main(String[] args) { Helper.run(); }
}

// Helper.java
class Helper {
    static void run() { System.out.println("Hello!"); }
}
```

Khi bạn chạy lệnh `java Prog.java`, trình khởi chạy Java sẽ biên dịch và thực thi phương thức `main` của lớp `Prog` trong bộ nhớ. Do code trong lớp `Prog` tham chiếu đến lớp `Helper`, trình khởi chạy sẽ tự động tìm tệp `Helper.java` trên hệ thống tệp, biên dịch lớp `Helper`, và thực thi nó trong bộ nhớ. Quá trình này diễn ra tự động, nhà phát triển không cần gọi `javac` rõ ràng để biên dịch tất cả các tệp nguồn.

Tính năng này làm cho việc chuyển đổi từ dự án nhỏ sang dự án lớn trở nên mượt mà hơn, nhà phát triển có thể tự do lựa chọn khi nào nên giới thiệu công cụ build, tránh phải thiết lập cấu trúc dự án phức tạp khi đang lặp lại nhanh. Tính năng này loại bỏ giới hạn đơn tệp, tiếp tục đơn giản hóa quá trình phát triển từ chương trình đơn tệp đến đa tệp, đặc biệt phù hợp cho phát triển nguyên mẫu, thử nghiệm nhanh và giai đoạn khám phá đầu của dự án.
