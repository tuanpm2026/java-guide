### JEP 305: Pattern Matching for instanceof (Pattern Matching cho instanceof, Preview)

Java 14 tiếp tục đưa pattern matching cho instanceof làm preview feature, đây là tính năng được giới thiệu trong Java 14 (JEP 305).

Tính năng này cho phép thực hiện type casting đồng thời với kiểm tra instanceof, tránh việc explicit type casting. Lưu ý: instanceof pattern matching là preview lần đầu trong Java 14 (JEP 305), preview lần hai trong Java 15 (JEP 375), cuối cùng chính thức trong Java 16 (JEP 394).

### JEP 358: Helpful NullPointerExceptions (Thông báo lỗi NullPointerException chính xác)

Thêm `-XX:+ShowCodeDetailsInExceptionMessages` vào JVM params để lấy thông tin call chi tiết hơn trong NullPointerException, giúp định vị và giải quyết vấn đề nhanh hơn.

```java
a.b.c.i = 99; // Giả sử đoạn code này gây ra NullPointerException
```

Trước Java 14:

```java
Exception in thread "main" java.lang.NullPointerException
    at NullPointerExample.main(NullPointerExample.java:5)
```

Từ Java 14:

```java
 // Sau khi thêm param, exception message chỉ rõ chỗ nào null gây ra lỗi
Exception in thread "main" java.lang.NullPointerException:
        Cannot read field 'c' because 'a.b' is null.
    at Prog.main(Prog.java:5)
```

### JEP 361: Switch Expressions (Switch Expressions, Standard)

Switch (preview feature) được giới thiệu trong Java 12, trở thành phiên bản chính thức trong Java 14, không cần thêm params để bật, có thể dùng trực tiếp trong JDK 14.

Java 12 giới thiệu lambda-like syntax cho switch expressions khi điều kiện match thành công, không cần thêm break; Java 13 cung cấp `yield` để trả về giá trị trong block.

```java
String result = switch (day) {
            case "M", "W", "F" -> "MWF";
            case "T", "TH", "S" -> "TTS";
            default -> {
                if(day.isEmpty())
                    yield "Please insert a valid day.";
                else
                    yield "Looks like a Sunday.";
            }

        };
System.out.println(result);
```

### JEP 359: Records (Record Class, Preview)

Từ khóa `record` có thể đơn giản hóa cách định nghĩa **data classes** (class Java khi đã được khởi tạo thì không thể thay đổi nữa). Dùng `record` thay cho `class` để định nghĩa class, chỉ cần khai báo attributes là tự động có access methods, `toString()`, `hashCode()`, `equals()`.

Tương tự như dùng `class` để định nghĩa class, đồng thời dùng lombok plugin với các annotation `@Getter,@ToString,@EqualsAndHashCode`.

```java
/**
 * Class này có hai đặc điểm
 * 1. Tất cả member attributes đều là final
 * 2. Toàn bộ methods bao gồm constructor và hai accessor methods cho member attributes (tổng cộng 3)
 * Loại class này rất phù hợp để khai báo bằng record
 */
final class Rectangle implements Shape {
    final double length;
    final double width;

    public Rectangle(double length, double width) {
        this.length = length;
        this.width = width;
    }

    double length() { return length; }
    double width() { return width; }
}
/**
 * 1. Class được khai báo bằng record sẽ tự động có ba methods của class trên
 * 2. Ngoài ra còn được tặng thêm equals(), hashCode() và toString()
 * 3. toString method bao gồm string representation và tên của tất cả member attributes
 */
record Rectangle(float length, float width) { }
```

### JEP 368: Text Blocks (Text Blocks, Preview Lần 2)

Trong Java 14, text blocks vẫn là preview feature, nhưng đã giới thiệu hai escape characters mới:

- `\` : Biểu thị end of line, không tạo newline character
- `\s`: Biểu thị single space

```java
String str = "Lòng hướng về đâu, bước chân theo đó, sống như nước ngược, một cành lau mà vượt.";

String str2 = """
        Lòng hướng về đâu,素履所往， \
        生如逆旅，一苇以航。""";
System.out.println(str2);// Lòng hướng về đâu, 素履所往， 生如逆旅，一苇以航。
String text = """
        java
        c++\sphp
        """;
System.out.println(text);
//Output:
java
c++ php
```

### JEP 379: Shenandoah (Shenandoah GC, Standard)

Shenandoah garbage collector được giới thiệu là experimental feature từ Java 12, sau nhiều phiên bản iteration và hoàn thiện, cuối cùng chính thức trong Java 15.

Shenandoah là low-latency garbage collector do Red Hat phát triển, mục tiêu chính:

- 99.9% GC pause time nhỏ hơn 10ms
- Pause time không phụ thuộc vào kích thước heap
- Hỗ trợ heap memory từ vài trăm MB đến vài TB

Khác với G1 và ZGC, Shenandoah dùng concurrent mark-compact algorithm, có thể thực hiện garbage collection mà không cần dừng application threads.

Bật Shenandoah GC:

```bash
java -XX:+UseShenandoahGC className
```

### JEP 339: EdDSA (Thuật toán chữ ký số)

Bổ sung thuật toán chữ ký số mới được implement dựa trên Edwards-Curve Digital Signature Algorithm (EdDSA) với cả security và performance đều tốt hơn.

Mặc dù performance vượt trội hơn implementation ECDSA hiện tại, nhưng nó không hoàn toàn thay thế Elliptic Curve Digital Signature Algorithm (ECDSA) hiện có trong JDK.

```java
KeyPairGenerator kpg = KeyPairGenerator.getInstance("Ed25519");
KeyPair kp = kpg.generateKeyPair();

byte[] msg = "test_string".getBytes(StandardCharsets.UTF_8);

Signature sig = Signature.getInstance("Ed25519");
sig.initSign(kp.getPrivate());
sig.update(msg);
byte[] s = sig.sign();

String encodedString = Base64.getEncoder().encodeToString(s);
System.out.println(encodedString);
```

Output:

```plain
0Hc0lxxASZNvS52WsvnncJOH/mlFhnA8Tc6D/k5DtAX5BSsNVjtPF4R4+yMWXVjrvB2mxVXmChIbki6goFBgAg==
```

### JEP 371: Hidden Classes (Hidden Classes)

Hidden classes được thiết kế cho frameworks, hỗ trợ dynamically generated classes, các classes này không thể được sử dụng trực tiếp bởi bytecode của các classes khác, chỉ có thể dùng gián tiếp qua reflection tại runtime.

Đặc điểm chính:

- Không thể phát hiện: Hidden classes không thể được depend trực tiếp bởi các classes khác
- Lifetime ngắn: Thường được unload sau khi dùng xong
- Performance optimization: Cung cấp performance support tốt hơn cho dynamic language runtimes

Use cases:

- Dynamic language runtimes (như JavaScript, Groovy)
- Bytecode generation frameworks (như ASM, CGLIB)
- Reflection proxies (như dynamic proxy)

### JEP 360: Sealed Classes (Sealed Classes, Preview)

**Sealed Classes** là một preview feature mới trong Java 15.

Trước khi có sealed classes, trong Java nếu muốn một class không thể bị kế thừa và sửa đổi, ta dùng từ khóa `final`. Tuy nhiên, cách này không linh hoạt, chặn hoàn toàn đường kế thừa và sửa đổi của class.

Sealed classes có thể giới hạn các classes được kế thừa hoặc implement chúng, như vậy class chỉ có thể được kế thừa bởi các classes cụ thể được chỉ định.

```java
// Abstract class Person chỉ cho phép Employee và Manager kế thừa.
public abstract sealed class Person
    permits Employee, Manager {

    //...
}
```

Ngoài ra, bất kỳ class nào extends sealed class đều phải khai báo `sealed`, `non-sealed` hoặc `final`.

```java
public final class Employee extends Person {
}

public non-sealed class Manager extends Person {
}
```

![](https://oss.javaguide.cn/javaguide/image-20210820153955587.png)

Nếu subclasses được phép extend và sealed class nằm trong cùng source code file, sealed class có thể không dùng lệnh permits, Java compiler sẽ tìm trong source file và tự động thêm các permitted subclasses vào sealed class lúc compile.

### JEP 375: Pattern Matching for instanceof (instanceof Pattern Matching, Preview Lần 2)

Java 15 tiếp tục đưa instanceof pattern matching làm preview feature, không có điều chỉnh lớn, chủ yếu để thu thập thêm phản hồi sử dụng.

instanceof pattern matching cho phép binding variable đồng thời với type check, tránh explicit type conversion, làm code gọn gàng và an toàn hơn.

Ví dụ:

```java
// Traditional approach
if (obj instanceof String) {
    String str = (String) obj;
    System.out.println(str.length());
}

// Pattern matching approach
if (obj instanceof String str) {
    System.out.println(str.length());
}
```

### API Enhancements

#### CharSequence Enhancement

Trước Java 15, nếu muốn kiểm tra `StringBuilder`, `StringBuffer` hay `CharBuffer` có rỗng không, thường phải gọi `length() == 0`.

Interface `CharSequence` được bổ sung default method `isEmpty()` để kiểm tra chuỗi ký tự có rỗng không, trả về true nếu rỗng.

```java
public interface CharSequence {
  default boolean isEmpty() {
      return this.length() == 0;
  }
}
```

Vì `String`, `StringBuilder`... đều implement interface này, bây giờ có thể viết logic kiểm tra theo cách thống nhất và có ngữ nghĩa hơn. Điều này rất thân thiện với code generic (xử lý các loại character sequences khác nhau).

**Lưu ý:** Class `String` đã có method `isEmpty()` từ lâu, nhưng đó là `String` tự định nghĩa; bước này trong Java 15 là "đưa" khả năng đó lên interface level.

#### TreeMap Performance Improvement

Đây là một tối ưu hóa rất hardcore. Mặc dù các methods như `putIfAbsent()`, `compute()`... đã xuất hiện trong interface `Map` từ Java 8, nhưng trước Java 15, `TreeMap` chưa tự implement chúng mà trực tiếp dùng implementation `default` trong interface.

**Thay đổi trong Java 15**: `TreeMap` đặc biệt override các methods sau:

- `putIfAbsent()`
- `computeIfAbsent()`
- `computeIfPresent()`
- `compute()`
- `merge()`

**Tại sao phải override?**

- **Performance improvement**: Default implementation trong interface thường khá "ngu ngốc", thường cần nhiều lần lookup (ví dụ tìm xem đã tồn tại chưa, sau đó mới quyết định có insert không).
- **O(log n) guarantee**: Implementation được tối ưu của `TreeMap` có thể gộp lookup và insert trong một lần duyệt red-black tree, tránh tree search lặp lại, cải thiện đáng kể hiệu suất khi xử lý large-scale data.
