## Java 12

JDK 12 được phát hành vào ngày 19 tháng 3 năm 2019, đây là phiên bản non-LTS.

Bài viết này sẽ chọn lọc một số tính năng mới quan trọng hơn để giới thiệu chi tiết:

- [JEP 189: Shenandoah: A Low-Pause-Time Garbage Collector (Experimental)](https://openjdk.org/jeps/189)
- [JEP 325: Switch Expressions (Preview) (biểu thức switch, tính năng xem trước)](https://openjdk.org/jeps/325)
- [JEP 334: JVM Constants API (API hằng số JVM)](https://openjdk.org/jeps/334)
- [JEP 344: Abortable Mixed Collections for G1 (G1 tập hợp hỗn hợp có thể hủy)](https://openjdk.org/jeps/344)
- [JEP 346: Promptly Return Unused Committed Memory (G1 trả về kịp thời bộ nhớ đã cấp phát chưa sử dụng)](https://openjdk.org/jeps/346)

Hình dưới đây là số lượng tính năng mới và thời gian cập nhật của mỗi phiên bản từ JDK 8 đến JDK 25:

![ Số lượng tính năng mới và thời gian cập nhật của mỗi phiên bản từ JDK 8 đến JDK 25](https://oss.javaguide.cn/github/javaguide/java/new-features/jdk8~jdk24.png)

### JEP 189: Shenandoah (Bộ thu gom rác độ trễ thấp, thử nghiệm)

Triển khai Pauseless GC do Redhat dẫn dắt phát triển, mục tiêu chính là 99.9% các lần dừng nhỏ hơn 10ms, thời gian dừng không phụ thuộc vào kích thước heap, v.v.

So với ZGC được mã nguồn mở trong Java 11 (cần nâng cấp lên JDK11 mới sử dụng được), Shenandoah GC có phiên bản JDK8u ổn định, có khả năng triển khai thực tế cao hơn trong bối cảnh Java 8 vẫn đang chiếm thị phần chủ yếu.

### JEP 344 & JEP 346: Tối ưu hóa bộ thu gom G1

Java 12 mang lại hai cập nhật cho bộ thu gom rác mặc định G1:

- **Tập hợp hỗn hợp có thể hủy**: Triển khai JEP 344, để đạt được mục tiêu thời gian dừng do người dùng cung cấp, JEP 344 chia tập hợp các vùng cần thu gom (tập hợp hỗn hợp) thành phần bắt buộc và phần tùy chọn, cho phép bộ thu gom rác G1 có thể hủy quá trình thu gom rác. G1 có thể hủy việc thu gom phần tùy chọn để đạt được mục tiêu thời gian dừng.
- **Trả về kịp thời bộ nhớ đã cấp phát chưa sử dụng**: Triển khai JEP 346, cải tiến G1 GC để tự động trả lại bộ nhớ heap Java cho hệ điều hành khi nhàn rỗi.

### JEP 334: JVM Constants API (API hằng số JVM)

Giới thiệu một API để mô hình hóa các mô tả danh nghĩa của các artifact file lớp và runtime quan trọng, đặc biệt là các hằng số có thể được tải từ constant pool.

API này cung cấp một tập hợp các interface và lớp tiện ích để biểu diễn và thao tác các mục constant pool trong file lớp. Nó chủ yếu bao gồm:

- **Interface mô tả hằng số**: Interface `ConstantDesc` và các sub-interface của nó, dùng để mô tả các loại hằng số khác nhau.
- **Loại giá trị hằng số**: `ClassDesc`, `MethodTypeDesc`, `MethodHandleDesc`, `DynamicConstantDesc`, v.v.
- **Bootstrap method**: Hỗ trợ lệnh `invokedynamic` và bootstrap method hằng số động.

API này chủ yếu được thiết kế để hỗ trợ các kịch bản sau:

1. **Thao tác file lớp**: Cung cấp cách thức chuẩn hóa để mô tả và thao tác constant pool trong file lớp.
2. **Tạo bytecode**: Đơn giản hóa tương tác giữa các framework tạo bytecode (như ASM) và code Java.
3. **Cải tiến reflection**: Làm cho các thao tác reflection an toàn kiểu hơn và có khả năng diễn đạt mạnh hơn.
4. **Công cụ compiler**: Cung cấp abstraction tốt hơn cho compiler và các công cụ tạo code.

API này là cải tiến nền tảng quan trọng trong Java 12, đặt nền móng cho các tính năng thao tác bytecode và compiler tiếp theo.

### JEP 325: Switch Expressions (biểu thức switch, xem trước)

Cú pháp `switch` truyền thống có vấn đề dễ quên viết `break`, và từ góc độ tính sạch của code, nhiều break về bản chất cũng là một dạng lặp lại.

Java 12 cải tiến biểu thức `switch`, sử dụng cú pháp tương tự lambda cho khối thực thi sau khi điều kiện khớp, không cần viết thêm break.

```java
switch (day) {
    case MONDAY, FRIDAY, SUNDAY -> System.out.println(6);
    case TUESDAY                -> System.out.println(7);
    case THURSDAY, SATURDAY     -> System.out.println(8);
    case WEDNESDAY              -> System.out.println(9);
}
```

## Java 13

JDK 13 được phát hành vào ngày 17 tháng 9 năm 2019, đây là phiên bản non-LTS.

Bài viết này sẽ chọn lọc một số tính năng mới quan trọng hơn để giới thiệu chi tiết:

- [JEP 350: Dynamic CDS Archives (lưu trữ CDS động)](https://openjdk.org/jeps/350)
- [JEP 351: ZGC: Uncommit Unused Memory (ZGC giải phóng bộ nhớ chưa sử dụng)](https://openjdk.org/jeps/351)
- [JEP 355: Text Blocks (Preview) (khối văn bản, tính năng xem trước)](https://openjdk.org/jeps/355)
- [JEP 354: Switch Expressions (Second Preview) (biểu thức switch, lần xem trước thứ hai)](https://openjdk.org/jeps/354)

Hình dưới đây là số lượng tính năng mới và thời gian cập nhật của mỗi phiên bản từ JDK 8 đến JDK 24:

![](https://oss.javaguide.cn/github/javaguide/java/new-features/jdk8~jdk24.png)

### JEP 351: ZGC (giải phóng bộ nhớ chưa sử dụng)

ZGC được giới thiệu thử nghiệm trong Java 11 có vấn đề không chủ động trả lại bộ nhớ chưa sử dụng cho hệ điều hành trong thực tế sử dụng.

Heap ZGC bao gồm một tập hợp các vùng heap được gọi là ZPages. Khi các vùng ZPages được làm trống trong chu kỳ GC, chúng sẽ được giải phóng và trả về bộ nhớ đệm trang **ZPageCache**. Các ZPage trong bộ nhớ đệm này được sắp xếp theo thứ tự ít được sử dụng gần đây nhất (LRU) và theo kích thước.

Trong Java 13, ZGC sẽ trả lại cho hệ điều hành các trang được xác định là chưa được sử dụng trong thời gian dài, để chúng có thể được tái sử dụng bởi các tiến trình khác.

### JEP 350: Dynamic CDS Archives (lưu trữ CDS động)

Java 13 đã đơn giản hóa, cải tiến và mở rộng thêm AppCDS (Application Class Data Sharing) được giới thiệu trong Java 10, cụ thể là: **cho phép lưu trữ lớp động khi ứng dụng Java kết thúc thực thi**. Các lớp có thể được lưu trữ bao gồm tất cả các lớp đã được tải nhưng không thuộc CDS cơ sở mặc định — gồm các lớp ứng dụng và các lớp trong thư viện tham chiếu.

Điều này cải thiện tính khả dụng của Application Class Data Sharing ([AppCDS](https://openjdk.java.net/jeps/310)). Không cần người dùng thực hiện chạy thử để tạo danh sách lớp cho từng ứng dụng.

```bash
java -XX:ArchiveClassesAtExit=my_app_cds.jsa -cp my_app.jar
java -XX:SharedArchiveFile=my_app_cds.jsa -cp my_app.jar
```

### JEP 355: Text Blocks (khối văn bản, xem trước)

Giải quyết vấn đề Java phải dùng escape ký tự xuống dòng hoặc toán tử nối dòng để định nghĩa chuỗi nhiều dòng, bằng cách giới thiệu **ba dấu nháy kép** để định nghĩa văn bản nhiều dòng.

Java 13 hỗ trợ bất kỳ nội dung nào nằm giữa hai ký hiệu `"""` sẽ được hiểu là một phần của chuỗi, bao gồm cả ký tự xuống dòng. Lưu ý: "hai" ở đây nên hiểu là "một cặp", tức là một ký hiệu mở đầu và một ký hiệu kết thúc.

Cách viết HTML trước khi hỗ trợ khối văn bản:

```java
String json ="{\n" +
              "   \"name\":\"mkyong\",\n" +
              "   \"age\":38\n" +
              "}\n";
```

Cách viết HTML sau khi hỗ trợ khối văn bản:

```java
 String json = """
                {
                    "name":"mkyong",
                    "age":38
                }
                """;
```

Cách viết SQL trước khi hỗ trợ khối văn bản:

```sql
String query = "SELECT `EMP_ID`, `LAST_NAME` FROM `EMPLOYEE_TB`\n" +
               "WHERE `CITY` = 'INDIANAPOLIS'\n" +
               "ORDER BY `EMP_ID`, `LAST_NAME`;\n";
```

Cách viết SQL sau khi hỗ trợ khối văn bản:

```sql
String query = """
               SELECT `EMP_ID`, `LAST_NAME` FROM `EMPLOYEE_TB`
               WHERE `CITY` = 'INDIANAPOLIS'
               ORDER BY `EMP_ID`, `LAST_NAME`;
               """;
```

Phần giới thiệu các phương thức liên quan đến khối văn bản (`formatted()`, `stripIndent()`, `translateEscapes()`) vui lòng xem phần [Cải tiến API - Cải tiến String (các phương thức liên quan đến khối văn bản)](#string-增强文本块相关方法) trong bài này.

### JEP 354: Switch Expressions (biểu thức switch, lần xem trước thứ hai)

Biểu thức `Switch` có thêm một từ khóa `yield` để thoát khỏi khối `Switch`, chủ yếu dùng để trả về một giá trị.

Sự khác biệt giữa `yield` và `return` là: `return` sẽ thoát trực tiếp khỏi vòng lặp hoặc phương thức hiện tại, còn `yield` chỉ thoát khỏi khối `Switch` hiện tại. Đồng thời khi sử dụng `yield`, cần có điều kiện `default`.

```java
 private static String descLanguage(String name) {
        return switch (name) {
            case "Java": yield "object-oriented, platform independent and secured";
            case "Ruby": yield "a programmer's best friend";
            default: yield name +" is a good language";
        };
 }
```

## Cải tiến API

Không phải tất cả các thay đổi API đều được phát hành thông qua JEP (Java Enhancement Proposal).

Trong quy trình phát triển JDK: **JEP** thường được dùng cho các thay đổi lớn, chẳng hạn như giới thiệu các tính năng ngôn ngữ mới (như biểu thức `switch`), cơ chế JVM mới (như ZGC) hoặc tái cấu trúc thư viện quy mô lớn. Các thao tác như thêm một vài phương thức vào lớp hiện có kiểu `String.indent()` thường được coi là bảo trì thư viện thông thường. Chúng được các nhà phát triển JDK trực tiếp gửi và review qua ticket **JBS (JDK Bug System)**, sau đó phát hành trực tiếp theo phiên bản.

### Cải tiến String

Java 12 thêm hai phương thức xử lý chuỗi mới.

#### indent() - Phương thức thụt lề

Phương thức `indent()` có thể thực hiện thụt lề chuỗi.

```java
String text = "Java";
// Thụt lề 4 ký tự
text = text.indent(4);
System.out.println(text);
text = text.indent(-10);
System.out.println(text);
```

Đầu ra:

```plain
     Java
Java
```

#### transform() - Phương thức chuyển đổi

Phương thức `transform()` có thể dùng để chuyển đổi chuỗi được chỉ định.

```java
String result = "foo".transform(input -> input + " bar");
System.out.println(result); // foo bar
```

### Cải tiến Files

Java 12 thêm phương thức `mismatch()` để so sánh hai file:

```java
public static long mismatch(Path path, Path path2) throws IOException
```

Phương thức `mismatch()` dùng để so sánh hai file và trả về vị trí ký tự đầu tiên không khớp, trả về -1L nếu các file giống nhau.

Ví dụ code (trường hợp nội dung hai file giống nhau):

```java
Path filePath1 = Files.createTempFile("file1", ".txt");
Path filePath2 = Files.createTempFile("file2", ".txt");
Files.writeString(filePath1, "Java 12 Article");
Files.writeString(filePath2, "Java 12 Article");

long mismatch = Files.mismatch(filePath1, filePath2);
assertEquals(-1, mismatch);
```

Ví dụ code (trường hợp nội dung hai file khác nhau):

```java
Path filePath3 = Files.createTempFile("file3", ".txt");
Path filePath4 = Files.createTempFile("file4", ".txt");
Files.writeString(filePath3, "Java 12 Article");
Files.writeString(filePath4, "Java 12 Tutorial");

long mismatch = Files.mismatch(filePath3, filePath4);
assertEquals(8, mismatch);
```

### Cải tiến NumberFormat

Java 12 bổ sung hỗ trợ định dạng số phức tạp cho `NumberFormat`:

```java
NumberFormat fmt = NumberFormat.getCompactNumberInstance(Locale.US, NumberFormat.Style.SHORT);
String result = fmt.format(1000);
System.out.println(result);
```

Đầu ra:

```plain
1K
```

### Cải tiến Socket API

Java 13 đã viết lại nền tảng của Socket API, `NioSocketImpl` là sự thay thế trực tiếp cho `PlainSocketImpl`, nó sử dụng lock trong package `java.util.concurrent` thay vì các phương thức đồng bộ hóa. Nếu muốn sử dụng triển khai cũ, hãy dùng `-Djdk.net.usePlainSocketImpl=true`.

Và trong Java 13, triển khai Socket mới được sử dụng mặc định.

```java
public final class NioSocketImpl extends SocketImpl implements PlatformSocketImpl {
}
```

### Cải tiến FileSystems

Java 13 thêm ba phương thức mới trong lớp `FileSystems`, để dễ dàng sử dụng hơn với các nhà cung cấp hệ thống file xem nội dung file như một hệ thống file:

- `newFileSystem(Path)`
- `newFileSystem(Path, Map<String, ?>)`
- `newFileSystem(Path, Map<String, ?>, ClassLoader)`

### Cải tiến String (các phương thức liên quan đến khối văn bản)

Java 13 giới thiệu tính năng xem trước Text Blocks (khối văn bản), lớp `String` thêm 3 phương thức mới để thao tác với khối văn bản:

- `formatted(Object... args)`: Tương tự phương thức `format()` của `String`. Được thêm vào để hỗ trợ định dạng khối văn bản.
- `stripIndent()`: Dùng để loại bỏ khoảng trắng ở đầu và cuối mỗi dòng trong khối văn bản.
- `translateEscapes()`: Chuyển đổi chuỗi escape như _"\\\t"_ thành _"\t"_

Do khối văn bản là tính năng xem trước, có thể bị xóa trong các phiên bản tương lai, nên các phương thức mới này được đánh dấu là deprecated.

```java
@Deprecated(forRemoval=true, since="13")
public String stripIndent() {
}
@Deprecated(forRemoval=true, since="13")
public String formatted(Object... args) {

}
@Deprecated(forRemoval=true, since="13")
public String translateEscapes() {
}
```

Để biết thêm chi tiết về khối văn bản, vui lòng xem phần [JEP 355: Text Blocks (Preview)](#jep-355-text-blocks-preview) trong bài này.

## Bổ sung

### Về tính năng xem trước

Trích dẫn nguyên văn từ trang web chính thức của Oracle: `This is a preview feature, which is a feature whose design, specification, and implementation are complete, but is not permanent, which means that the feature may exist in a different form or not at all in future JDK releases. To compile and run code that contains preview features, you must specify additional command-line options.`

Đây là tính năng xem trước, tức là tính năng có thiết kế, thông số kỹ thuật và triển khai hoàn chỉnh, nhưng không phải là vĩnh viễn. Điều này có nghĩa là tính năng này có thể tồn tại ở dạng khác hoặc hoàn toàn không tồn tại trong các phiên bản JDK tương lai. Để biên dịch và chạy code chứa các tính năng xem trước, bạn phải chỉ định thêm các tùy chọn dòng lệnh.

Lấy việc cải tiến `switch` làm ví dụ, từ khi ra mắt trong Java 12, tiếp tục được cải tiến trong Java 13, cho đến Java 14 mới chính thức được chấp thuận vào JDK và có thể sử dụng an tâm mà không cần lo lắng về việc sửa đổi hoặc thay đổi trong các phiên bản JDK tiếp theo.

Một mặt có thể thấy thái độ nghiêm túc của JDK với tư cách là nền tảng chuẩn khi bổ sung các tính năng mới, mặt khác cá nhân tôi cho rằng nên có thái độ thận trọng khi sử dụng các tính năng xem trước. Thiết kế và triển khai tính năng thì dễ, nhưng giá trị thực tế của nó vẫn cần được kiểm chứng trong quá trình sử dụng.

### Tối ưu hóa JVM

Mỗi lần phát hành phiên bản Java đều đi kèm với tối ưu hóa JVM, bao gồm cải tiến các thuật toán thu gom rác hiện có, giới thiệu các thuật toán thu gom rác mới, loại bỏ các thuật toán thu gom rác cũ không còn phù hợp với ngày nay.

Hướng tối ưu hóa tổng thể là **hiệu suất thu gom rác cao, độ trễ thấp**.

Các nhà phát triển ứng dụng hàng ngày có thể quan tâm nhiều hơn đến các tính năng cú pháp mới, nhưng từ góc độ của một công ty, khi xem xét có nên nâng cấp nền tảng Java hay không, điều được cân nhắc nhiều hơn là **sự cải tiến của JVM runtime**.
